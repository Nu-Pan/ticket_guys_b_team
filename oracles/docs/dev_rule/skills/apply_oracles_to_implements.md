# スキル `apply-oracles-to-implements`

## 概要

- `<tgbt-root>/oracles` の内容を実装に反映させる能力をパッケージングしたスキル
- 「`<tgbt-root>/oracles` を実装に反映させる」の何たるかを明文化し、それを人間が毎回入力しなくて済むようにする事を狙う

## スキルの目的

- 目的とする
    - `<tgbt-root>/oracles` の記述と実装との間に矛盾があった場合 `<tgbt-root>/oracles` を正本として扱い、実装側を修正する
    - 作業の過程で発見した、実装の致命的な問題の修正
    - 根本問題の解決のために連鎖的に発生する問題の修正
    - `<tgbt-root>/oracles` の内容から考えて、不要と思われる実装の削除
    - 必要な作業の結果としての、実装の品質改善
    - 作業完了後、人間が適切なフォローアップ作業を作成可能であること
- 目的としない
    - リファクター・品質改善そのものを目的（ゴール）とした作業

## `<tgbt-root>/oracles` の調査範囲

- スキル外からの人間指示がない場合
    - 暗黙に「直近の変更内容に基づいた実装追従作業」であると仮定する
    - 具体的には、 git の「未コミットの変更内容」「直近数コミットの変更内容」に基づいて、実際の意味論的な変更内容を調査する
- スキル外からの人間指示がある場合
    - e.g. 変更したファイルのパスを提示、変更した内容の概要を記述、全体の再チェックを依頼
    - この場合は人間からの指示に従う

## 自己レビュー

### 前提

- 実際の AI による作業は「実装計画」「実装作業」の２フェーズに分かれるものとする
    - これは `/plan` モードとの併用を想定している
- それぞれのフェーズについて、完了前に AI による自己レビューを行うものとする
- 最低限含めるべき具体的な自己レビュー観点を以下に述べる
    - あくまで最低限であり AI の裁量でレビュー観点を増やしても良いものとする

### 実装計画レビュー観点

- oracles の変更内容、または人間から指定された oracles 参照範囲が、実装計画内で具体的な実装対象へ対応付けられているか？
- oracles と実装・既存 state・ユーザー指示が衝突している場合、oracles を正本として扱い、衝突内容を隠さず明示しているか？
- 計画の対象が、関数・クラス・コマンド・入出力など、AI が扱うべき low-level な実装単位へ十分に局所化されているか？
- high-level な仕様判断、public interface 変更、依存追加、永続データ形式変更、大規模リファクタが必要な場合に、人間確認事項として切り出せているか？
- 既存設計・既存責務・既存 import 構造に沿う方針になっており、不要な新規抽象化や責務移動を含んでいないか？
- 変更対象ファイルと読み取り対象ファイルが必要最小限に絞られているか？
- `<tgbt-root>/README.md`、`<tgbt-root>/memo/**`、`<tgbt-root>/oracles/**` など、編集・閲覧制約に反する作業を計画していないか？
- tgbt 自体の開発として、テスト追加・更新を前提にしていないか？
- 実装後の確認方法が、変更対象に近い `pyright`、import 確認、必要最小限の smoke 確認などに落ちているか？
- 不確実性、未確認事項、連鎖的に発生しうる問題が、計画のリスクまたは人間確認事項として明示されているか？

### 実装作業レビュー観点

- 実装結果が oracles の明示内容と矛盾していないか？
- oracles と実装の衝突を解消するために、oracles 側を編集したり、oracles の意味を勝手に読み替えたりしていないか？
- oracles に未記載の仕様について、既存実装と局所文脈から必要最小限の判断に留めているか？
- 要求を満たす上で必要のない余計なリファクタ、整形、責務移動、public interface 変更、依存追加、永続データ形式変更が混ざっていないか？
- 作業中に発見した致命的な実装問題、または根本問題の解決に必要な連鎖的問題だけを追加修正対象にしているか？
- 既存の呼び出し元、保存形式、エラー処理、ログ、prompt/schema の意味論を不必要に変更していないか？
- 各 `oracles` で述べられている原則・規則に違反していないか
    - 実際の `SKILL.md` では参照するべき `oracles` 配下の `.md` を提示する
    - e.g. python コーディングについては `<tgbt-root>/oracles/docs/dev_rule/python_coding.md` 
- 確認で出たエラーを、要求範囲内で修正しているか？ 範囲外の既存エラーは、修正せずに人間へ報告できているか？

## 作業レポート

- Codex CLI の通常の作業完了報告に加えて「人間が適切なフォローアップ作業を作成するために必要な情報」を報告すること
- 具体的には、最低でも以下の内容が必要
    - 未解決の問題点・リスク
    - 人間判断が無いと実行に移せないこと
    - 参照した oracles の記述と、それに対応して変更した実装箇所
    - 作業中に発見した問題のうち、今回修正したもの・修正しなかったものの区別
    - 実行した確認内容と結果、または実行できなかった確認とその理由
    - 人間が次に作成するとよさそうなフォローアップ作業候補
