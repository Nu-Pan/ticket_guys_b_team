## docs を消す？

### プラン A

- 方針
    - ドキュメントは oracle だけにして AI によるドキュメント整理はやらせない
    - 作業ごとに毎回調べ直させるということ
- 必要作業
    - そのためには docs の内容を oracle に引き上げる必要がある

### プラン B

- ドキュメントは oracle, docs の２段構え
- docs は「実装に対する調査結果のキャッシュ」と位置づける
    - docs 無しだと AI が新規作業に着手するごとに毎回ソースファイルを見に行く必要がある
    - ソースコードは調査用の情報ではないため、要約作業が必要
    - 要約作業をするにはソースを一度読む必要があるのでコンテキストを圧迫する
    - 要約後の情報をセッション間で情報を持ち越す仕組みがあれば、コンテキスト圧力を下げられる
- キャッシュをの陳腐化対策
    - キャッシュと言うことは、当然陳腐化が問題になる
    - 不正確な情報は削除（cache-out）する方向性でクリーンナップすれば docs の無限肥大化をさけられるか？
    - docs の記述には必ず対応するコードの位置情報を併記するのは確実に必要だろう

### プラン C

- ある plan に紐づいた「スクラッチバッファ的なファイル」を１つ用意
    - チケット間で情報を受け渡すための領域
- スクラッチバッファは AI 自身に管理させる
    - チケットの実行完了時に、計画・後続チケットから考えて残すべき情報をスクラッチバッファに書かせるとか
    - スクラッチバッファに書いてある事と現実との整合性を確認させるとか
    - そういったメンテナンス作業を tgbt から明示的に流し込む

### 考察

- garbage-in garbage-out は明確な懸念点
    - AI 編集前提の docs を用意する場合、 docs の内容が正確であるという保証が必要
    - 仮に docs の内容が不正確だとすると「不正確な情報 --> 不正確な計画 --> 間違った作業 --> それに対するリカバリ」みたいな感じで余計な労力が発生する
    - むしろ、 docs に書き出させれば AI による自己検証が可能なので、 docs 前提のほうが情報の正確性を保証しやすいかも
- 消費トークン数（docs なし）
    - tgbt は長い計画を小さいチケットに細分化、チケットごとにセッションを立てる考え方
    - なので、docs を用意せずにチケットごとに毎回必ず調べ直すのは、消費トークン数的に不利か？
- 消費トークン数（docs あり）
    - 継ぎ足しで docs をキレイに保ち続けたほうが、有利？
    - docs の内容が正しいかのチェックすること自体にかなりのコストが掛かりそう
    - 例えば、 docs の正確性検証の結果、修正必要箇所が全体の１％だったとしても、検証自体には１００％分のコストが掛かる
    - スクリプトで事前に検証対象を枝刈りできる仕組みがあれば
    - Codex のハーネスエンジニアリング記事によれば「専用の仕組み」って言ってるので docs の維持にはそれ相応のコストがかかるのだろう
- docs で知見を使い回すこと自体の是非
    - チケットごとに立てるべき問いは違うはず
    - 使い回せるという前提が既に間違っているのでは？
    - 目的設定も無く「なんとなく情報を残してほしい」だと、 docs の内容がふわふわした内容になる危険性も
    - プランの存在を前提とすれば、どういう譲歩を残すのが適切なのかを考えるための材料が揃っているので、残る情報の質がまともになる可能性はある
    - スクラッチバッファの考え方は Codex CLI にある context のコンパクションと近いものがあるかもしれない
- キャッシュ陳腐化問題
    - どうやるにしても docs は実装のキャッシュ（別表現）なので、オリジナルが更新されればキャッシュは陳腐化する
    - 陳腐化した情報＝不正確な情報であり、これはとても有害なので、積極的に直す必要がある
    - そうなると、この陳腐化解消にトークン予算を回す必要がある
    - 不正確な情報を消す方向性のメンテなら、規模を小さく保ちやすいかもしれない
    - 「陳腐化対策 v.s. 毎回調べる」だと、実は余計なメンテナンス対象が存在しない分後者の方がマシなのでは？
    - なんなら docs ありだと、解くべき問題が増えて、余計なドツボにハマって、無駄にトークンを浪費する危険性も有る
    - 一方、小さいチケットを消化する過程でちょっとずつ状況が変わっても、それを正確にトラック出来るのは明確なメリット
- 知識管理・利用システムは１つの大きな問題と捉えた方が良い
    - これは「`docs/` のあり方の問題」じゃなくて「知識の管理・利用」の問題
    - 知識モジュールの設計問題として独立に議論できる
    - マークダウンファイルの１つ１つをごく小さな文章 Markdown front matter
