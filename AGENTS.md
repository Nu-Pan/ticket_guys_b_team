# 開発環境

## 基本

- WSL2 Ubuntu 24.04 on Windows 11
- vscode (with Remote Develpment Extension)
- `ticket_guy_b_team.code-workspace` を vscode を開いた環境
- Codex CLI が利用可能

## ファイルエンコード

- 「UTF-8 BOM なし」で統一する
- ツール都合がある場合は、例外的に「UTF-8 BOM なし」以外でも良しとする

## python

- python 3.12.3
- システムワイドの `python3` を直接使うのは禁止
- python 仮想環境として `.venv` を使う
    - python インタプリタは `.venv/bin/python` を使う
    - pip パッケージマネージャは `.venv/bin/python -m pip` を使う
- 仮想環境の管理操作
    - 仮想環境 `.venv` の新規作成
        ```
        /usr/bin/python3 -m venv .venv
        ```
    - 仮想環境 `.venv` へのパッケージインストール（権限昇格付きでの実行をユーザーに依頼すること）
        ```
        ./.venv/bin/python -m pip install -e ."[test]"
        ```
    - `.venv` への新規パッケージの追加
        - `pyporject.toml` に依存関係を追記する
        - 上記「仮想環境 `.venv` へのパッケージインストール」を実行

# コーディングにおける AI エージェント行動原則

## 基本

- 命名、責務、入出力を明確に保つこと
- コードの意味的な小さいブロックごとに、そのブロックで何をしているかのコメントを記入する
- コード単体ではその意図が伝わらないおそれがある場合は `NOTE` を付けてコメントを書く
- オーバーエンジニアリングは避ける
    - ユーザーからの指示を満たせる最小限度の変更だけを行う
    - コードを過剰に共通化しようとしない（処理の共通化自体は大事だが、やりすぎない）
    - 具体的な根拠を用意できないような「なんとなくこっちのほうが良い」はやらない

## python

- python コーディングスタイルはPEP 8 を遵守
- 型ヒント (type hint)
    - 必ず書く
    - 必要以上に `Any` を使わない
- パッケージ管理
    - 必要なパッケージが `.venv` に入っていない場合は `pyproject.toml` に設定を書いたうえで `.venv` を再生成する
    - それ以外のワークアラウンド的対応は禁止
- import のルール
    - `src/` をルートとした import が可能であることを前提とする
    - 可能ならば、 `.` を起点とする相対パスでの `import` を行う
    - 相対パスでの `import` が不可能な場合のみ、絶対パスでの `import` を行う
- `from __future__ import annotations` は禁止
    - 型チェッカー (pylance) と実行時の挙動を揃えるためのルールである
- `import` による循環参照の回避方法
    - モジュールの細分化・構造の調整による循環参照回避を試みること
    - どうしても回避不能な場合のみ `TYPE_CHECKING` を使っても良い
- 型チェックについて
    - コードの追記・修正後は `pyright` による型チェックを必ず行う
    - 型チェックで見つけたエラーは修正する
    - 型チェックは `.venv/bin/python -m pyright <file name>` で実行する
- docstring について
    - 全ての関数・クラス・…には docstring を書く
    - コードからだけでは読み取りにくい「その関数・クラス・…が一体何者であるのか」を意識して説明を書く
    - docstring は google style に従う
    - docstring が無駄に縦に長くならないように努力をする
    - 縦に短くするためであれば、書く意味の薄い情報 (e.g. シグネチャを読めばわかるような事しか書いてない `Args` とか) は省略しても構わない
- コメントについて
    - 基本的に日本語で書く
    - 変数名・型名などの識別子は日本語に訳さなくて良い
    - 英単語のままの方が意味が通じやすい場合は、無理に日本語に訳さず英単語のままで良い
- ログについて
    - ログに流すメッセージ（エラーとか）は英語で書く
    - これは、トラブルシューティング情報が文字コード由来のトラブを抱え込むリスクを回避するため

# テスト

## 基本

- `pytest` を使う
- テストの配置先は `tests/**/*.py`

## テストコード実装原則

- 個別のテストの行数が必要以上に嵩まないように努力すること
- そのためにテストのコンポーネント共通化・構造化を行うこと

## テスト戦略

- Codex CLI の利用有無で分かれる２系統のテストを用意する
- 「Codex CLI を使用するテスト」
    - 実際に Codex CLI を呼び出す
    - こちらは Codex CLI 呼び出し部分のテストだけを目的とする
    - テストによるトークン消費を限り無く押さえたいので、こちらのテストは可能な限り最小限度に留める
    - こちらは明示的に opt-in が指示された時だけ実行される
- 「ダミー実装を使用するテスト」
    - Codex CLI を呼び出す代わりに、あらかじめ決められた固定パターンを返すダミー実装を用いる
    - このダミー実装への切り替えは `tickget_guy_b_team` の機能（CLI のオプション引数）として用意する
    - テストの品質向上のために積極的に拡充・修正するのはこちら
    - 通常は、こちらのテストが通れば動作確認としては合格とする
