# 前提

- このリポジトリでは `ticket_guy_b_team` を開発する
- AI エージェントは、アプリケーション仕様そのものと、自分の作業ルールを混同しないこと
- アプリケーションの仕様は `doc/spec.md` を参照すること
- この `AGENTS.md` には、AI エージェントが`ticket_guy_b_team` の開発作業を進める時の行動原則だけを書く

# AI エージェント行動原則

## 作業前

- まず `README.md`、`doc/spec.md`、関連する `doc/` を読むこと
- 作業内容が `spec.md` と矛盾していないか確認すること
- 要求が曖昧な場合は、いきなり実装せず、仕様不足かチケット不足として扱うこと

## 作業中

- 変更は最小限にすること
- オーバーエンジニアリングを避けること
- 命名、責務、入出力を明確に保つこと
- コードだけで意図が伝わりにくい箇所には簡潔なコメントを書くこと
- テスト可能なものはテストし、静的検査可能なものは静的検査すること
- 機械的に判定できる検証を優先し、人間判断が必要な場合だけエスカレーションすること

## 作業完了後

- 変更に対応するテスト、静的検査、動作確認を行うこと
- 指示されていない失敗や制約があれば必ず報告すること
- 仕様不足、受け入れ条件不足、チケット分解不足で詰まった場合は、そのことを報告すること
- 作業中に発生した問題とその改善案があれば `doc/feedback.md` に記入すること

## レビュー原則

- 上流の仕様レビューを重視すること
- 下流は人間の全件レビューではなく、自動ゲート中心で考えること
- 人間レビューが必要なのは、仕様の正しさ、機械判定不能な例外、最終判断である

## 禁止事項

- 未承認仕様から直接実装を始めること
- 調査・仕様化・実装・検証を不必要に1つへ混ぜ込むこと
- 完了条件が曖昧なまま作業を進めること
- 自動検証なしで大量実行前提の設計を行うこと
- `AGENTS.md` を AI が編集すること
- 人間からの承認を得ずに `doc/feedback.md` の内容に手を付けること

# 開発環境

## 基本

- WSL2 Ubuntu 24.04 on Windows 11
- vscode (with Remote Develpment Extension)
- `ticket_guy_b_team.code-workspace` を vscode で開く
- Codex CLI が利用可能

## ファイルエンコード

- 「UTF-8 BOM なし」で統一する
- ツール都合がある場合は、例外的に「UTF-8 BOM なし」以外でも良しとする

## python

- python 3.12.3
- システムワイドの `python3` を直接使うことはしない
- python 仮想環境として `.venv` を使う
    - `.venv/bin/python` を使う
    - `.venv/bin/python -m pip` を使う
- `pyporject.toml` を書いて、それを `.venv/bin/python -m pip install -e .` でインストールする
- 必要なパッケージは `pyporject.toml` に書く

# コーディング

## AI エージェントコーディング原則

- コードの意味的な小さいブロックごとに、そのブロックで何をしているかのコメントを記入する
- コード単体ではその意図が伝わらないおそれがある場合は `NOTE` を付けてコメントを書く
- オーバーエンジニアリングは避ける
    - ユーザーからの指示を満たせる最小限度の変更だけを行う
    - コードを過剰に共通化しようとしない（処理の共通化自体は大事だが、やりすぎない）
    - 具体的な根拠を用意できないような「なんとなくこっちのほうが良い」はやらない

## python コーディング原則

- python コーディングスタイルはPEP 8 を遵守
- 型ヒント (type hint)
    - 必ず書いてください
    - 必要以上に `Any` を使わないようにしてください
- import の「深さ」のルール
    - あるモジュールを、そのモジュールの外から `import` する際は、モジュールのトップレベルだけを指定した深い `import` を使う
    - あるモジュールを、そのモジュールの中から `import` する際は、サブモジュールを指定した深い `import` を使う
- `from __future__ import annotations` は禁止
    - 型チェッカー (pylance) と実行時の挙動を揃えるためのルールである
- `import` による循環参照の回避方法が起きた場合
    - モジュールの細分化・構造の調整による循環参照回避を試みること
    - どうしても回避不能な場合は `TYPE_CHECKING` を使っても良い
- 型チェックについて
    - コードの追記・修正後は `pyright` による型チェックを必ず行ってください
    - 型チェックで見つけたエラーは修正してください
    - 型チェックは `.venv/bin/python -m pyright <file name>` で実行してください
- docstring について
    - 全ての関数・クラス・…には docstring を書いてください
    - コードからだけでは読み取りにくい「その関数・クラス・…が一体何者であるのか」を意識して説明を書いてください
    - docstring は google style に従ってください
    - docstring が無駄に縦に長くならないように努力をしてください
    - 縦に短くするためであれば、書く意味の薄い情報 (e.g. シグネチャを読めばわかるような事しか書いてない `Args` とか) は省略しても構いません
- コメントについて
    - 基本的に日本語で書く
    - 変数名・型名などの識別子は日本語に訳さなくて良い
    - 英単語のままの方が意味が通じやすい場合は、無理に日本語に訳さず英単語のままで良い
- ログについて
    - ログに流すメッセージ（エラーとか）は英語で書く
    - これは、トラブルシューティング情報が文字コード由来のトラブルで正しく表示されず問題を無駄に複雑化してしまうリスクを回避するため

## テストの方針

- `pytest` を使う
- テストの配置先
    - 典型的には `tests/**/*.py` という構成
- `pytest` 実装指針
    - 個別のテストの行数が必要以上に嵩まないように努力すること
- カバレッジ
    - `.venv/bin/python -m pytest --cov=...` を使ってカバレッジを計測可能
    - カバレッジ 90% を目標とする
