
# plan とは

- `tgbt plan ...` 系のサブコマンドで生成される作業計画書
- plan の内容（作業計画）に基づいて `tgbt run` が実行される
- 人間は plan を通して AI に対してディレクションを行う
- AI は（`tgbt run`は） plan による詳細なディレクションを前提に、長時間の自律的な作業を遂行する

# plan は AI によって編集される

- 人間は `tgbt plan ...` サブコマンド経由で間接的に plan の編集を行う
- 初回呼び出し時は、人間から与えられた指示文を元に AI が plan を新規に書き起こす
- ２回目以降の呼び出し時は、「人間から指定された plan id」＋「人間から与えられた指示文」を元に、既存 plan を加筆・修正する
- 「人間の脳内にだけある計画」と「AI が言語化した計画」との間で齟齬がなくなるまで plan の加筆・修正を繰り返す
- AI(Codex CLI) の呼び出しには `codex exec --output-schema` を使用し、構造化された応答を json として受け取って plan 正本へ反映する（AI による plan ファイルの直接編集は禁止）

# 計画書に含めるべき情報

- 人間から与えられたオリジナルの指示文（リスト）
- 作業の完了条件
- 計画（実行前）段階の想定作業手順
- AI 側で置いた仮定


# AI 側で仮定を置いて良いとする

- 「`tgbt plan ...` サブコマンド経由で与えられた指示」「`<repo-root>/oracle`で述べられた仕様断片」だけでは仕様は一意には定まらない
- よって「人間が明示していない隙間」は AI の仮定によって埋めても良しとする
- もしも仮定した内容が人間の意図にそぐわない場合、その事を `tgbt plan ...` サブコマンド経由で指摘するのは人間の責任である

# AI による plan 編集のフロー

## 基本的な考え方

- 人間が直接計画書を書き出すのは非常にコストが高いが、AI が書き出した計画を人間がチェックするのは比較的低コストという立場を取る
- この事から、 `tgbt plan` が人間に返す plan は AI による反復的な自己レビュー・修正を経た結果とする
- 自己レビューに合格した結果だけを人間に見せられるようベストエフォート的に努力するが、最悪の場合は未解決事項が残ったまま人間に返すことも許容する

## フロー

1. ユーザー指示に基づいてプランの新規作成・修正を実行
2. ループ
    1. plan の内容を機械に検査・修正
    2. AI による plan の内容のレビュー
    4. 機械的検査・レビューの内容に基づいて AI が plan を修正する
    4. 停止判定
3. 結果をレポート

## ユーザー指示に基づいてプランの新規作成・修正を実行

- `tgbt plan` コマンドでユーザーに指示された作業を行う
- まずは「普通に」作業を行うということ

## plan の内容を機械に検査・修正

- 非 AI の機械的検査を行う
- schema、ID、必須 field、空欄、重複、original_instructions、…
- 検査の結果、機械的な修正が可能であればそうする
- 機械的修正が不可能なもののみ、AI に対する指摘点としてリストアップする

## AI による plan の内容のレビュー

- 現状の plan に対するレビューを AI に行わせる
- レビュー結果は構造化出力で指摘リストとしてやり取りする
- レビューは「人間の意図を完全に当てる」ではなく、指示追従・明示性・実行可能性・リスク可視化を上げることを目的にする
- 合格・不合格の判定もこのステップ
    - ただし、 AI には指摘のメジャー・マイナー分類だけを行わせる
    - メジャー指摘が 0 件であれば、合格と判定する
- レビュー結果の構造化出力には専用のスキーマを用いる
- 少なくとも、以下のレビュー観点をプロンプトに注入する必要がある
    - completion_criteria が観測可能で、実行後に yes/no 判定できる
    - planned_procedures が順序付きで、1 item 1 action に近い粒度になっている
    - assumptions に、AI が埋めた仕様の隙間が明示されている
    - risk_notes に、曖昧さ、oracle との衝突、実行リスクが隠されず記録されている
    - user instruction と oracle が衝突する場合、oracle 優先で扱い、衝突を risk として残している
    - tgbt run がこの plan だけを読んで作業に入れる程度に、対象・手順・完了条件が具体化されている

## 機械的検査・レビューの内容に基づいて AI が plan を修正する

- そのループ内で出た指摘に対する修正作業を AI に依頼する
- 指摘が 0 件なら何もしない
- 入れ子ループはやらない
    - 修正セッション完了後、別セッションで作業結果を評価、不足があればもう一度修正セッションを起動…みたいな話はやらない
    - 修正作業に対する自己チェックはそのセッション内で完結させる（Codex CLI に押し付ける）

## 停止判定

- 以下のいずれかを満たしたらループ終了
    - 機械検査・AI レビュー両方に合格
    - ループ回数が一定回数（３～５回程度を想定）に達する

## 結果をレポート

- 機械検査・AI レビューの合格・不合格に関わらず、プランニングの結果を人間にレポートする
- 作成された最終的な plan ファイルのパスを `<repo-root>` からの相対パスでダンプする
- 不合格で終了した場合、その内容もをレポートに含める

## ログの扱い

- レビュー・修正の履歴は plan ファイルには混ぜない
- 内部的には AI エージェントラッパー (e.g. `CodexWrapper`) のログ機能で履歴を追跡でれば良しとする

# plan の実体・フォーマット

- 正本情報は json ファイル上に構造化された状態で永続化される
- 人間閲覧用に json から markdown をレンダリングし、それを `tgbt plan ...` の結果として返す

# plan 更新時の plan id の指定方法

- 既存 plan を更新する場合、引数として「編集したい plan の plan id」を渡す事を基本とする
- `tgbt plan` は必ず plan id をダンプするので、ユーザーは plan id を知っていると仮定して良い
- ただし、人間の操作負担軽減のため、plan id として latest が指定された場合は、既存 plan の中で最も新しい plan を選択する
