- このファイルでは「`tgbt` を使って任意のリポジトリ上で開発作業を行う」際の `oracle` について記述する
- 「`tgbt` リポジトリ上で Codex CLI を用いて `tgbt` 自体の開発を行う」際の `oracle` については `oracle/docs/dev_rule/oracle.md` を参照すること
- これら２パターンは根本的に前提が異なるため、混同してはいけない

# 基本的な考え方

- `<repo-root>/oracle` は AI による編集は禁止で人間が手動で真面目にメンテナンスする
- `oracle` に書いてる事があらゆる事柄の正本情報である
- `oracle` には網羅性は無く、人間が本当に重要だと思っていることだけが言語化されている
- 逆に言えば `oracle` に書いてないことは、人間からすると Don't Care であり、 AI による裁量が許される

# 狙い

- 人間がコストをかけて手動メンテナンスしなければならない領域を極小化することで
    - 手動メンテコストを人間の認知リソース予算内に収める
    - AI によるてこが効くようにする
- AI による編集を禁止することにより、人間意図のドリフティングを抑制する
- 「明文化された人間意図」と「それ以外」を分けることで、AI の責任・裁量境界を明確化する

# tgbt コマンドとの連携

- `tgbt` は、各コマンド経由で人間から与えられた指示を `<repo-root>/oracle` に自動で反映することはしない
- `tgbt eval oracle` サブコマンド
    - `<repo-root>/oracle` に対する評価を行い、その結果を人間にレポートする
    - 主に oracle のみで見た時の「矛盾の有無」「内容の簡略化の余地の有無」「文章構造最適化余地の有無」を評価する
    - ただし `oracle` の不足点は評価を行わない（oracle の記述は網羅的ではないことを前提とする）
- `tgbt plan` サブコマンド
    - 自然言語指示上で `<repo-root>/oracle` への参照を示すことはあり得るが、それ以上の特別扱いはしない
    - plan の指示と oracle の内容とで矛盾が発生した場合、それは計画書ドラフトに明記される

# テスト

- `<repo-root>/oracle/tests` を導入する
- `<repo-root>/oracle/tests` はここまでに述べた `<repo-root>/oracle` と同様の性質をもつ
- つまり、 AI による編集は禁止で、人間が１００％責任を持つ
- 人間が宣託を記述する方法として自然言語による指示が毎回最適とは限らないので、プログラムによる指示を行うインターフェースとしてテストを利用することを狙う
