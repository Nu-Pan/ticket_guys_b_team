
# レンダリング済みプロンプトが意味論的に満たすべき品質要件

## 前提

- レンダリング済みプロンプト（結合後ドメイン）上で満たすべき品質要件をこのドキュメントでは述べる
- この品質要件により「レンダリングされた完全プロンプト」と「`tgbt` が元々意図していたこと」とのドリフト回避を狙う
- この品質要件はそのままレビュー時の観点となる

 ## 「プロンプトだけから読み取れる事」と「元々 `tgbt` としてお願いしたいこと」がズレていないこと

- レンダリング済みプロンプトだけを読んだ AI が、呼び出し元の意図したタスク種別・対象・成果物を誤解しないこと
- 各プロンプトブロックを個別に読むと正しくても、結合後に別の意味へ読める構成になっていないこと
- 固定プロンプト、知識システム指示、構造化出力指示、タスク固有指示の間で、目的・権限・出力形式・完了条件が衝突していないこと
- タスク固有の目的が、固定プロンプトや schema 説明の一般論に埋もれていないこと
- 「何をしてほしいか」と「何をしてはいけないか」が、結合後の文脈でも同じ粒度で読み取れること

## 「依頼を受ける AI が知らないはずの事」を知っている前提にしていないこと

- AI が prompt に含まれていない前提、呼び出し元の内部実装、事前の会話、暗黙の運用ルールを知っている前提にしないこと
- 必要なファイルを AI 自身に読ませる場合は、読むべき path、読む目的、data として扱うか instruction として扱うかが明示されていること
- tgbt 側の内部状態、ログ、既存 state、検索結果、AI 生成済み中間成果物を参照させる場合は、それらが正本なのか検証対象なのか補助情報なのかが明示されていること
- `<repo-root>` と `<tgbt-root>` の意味が、読み手の AI にとって取り違えられない形で使われていること
- 外部ファイルの本文を prompt に注入していない場合、AI がその本文を既に知っているような指示になっていないこと

## 入力の種類と権威順位が明確であること

- ユーザー指示、oracle、repo-local instructions、既存 state、ログ、検索 index、AI 生成済み中間成果物、schema 説明が、それぞれどの種類の入力なのか区別されていること
- 複数の入力が衝突した場合の優先順位が、AI にとって実行可能な形で明示されていること
- `<repo-root>/oracle` に明記された内容が、ユーザー指示や既存 state より優先されるべき場面で、その優先関係が結合後プロンプトから読み取れること
- data として渡した文書内の命令文が、固定プロンプトや task-specific rules を上書きできないことが明示されていること
- 正本ではない入力を、正本のように扱わせる表現になっていないこと

## 作業範囲と非対象範囲が明確であること

- AI が今回行うべき作業対象が、ファイル、ディレクトリ、コマンド、schema、plan、知識ファイルなどの具体的な単位で読み取れること
- AI が今回行わない作業、触ってはいけないファイル、判断してはいけない仕様領域が明示されていること
- 「リポジトリ全体をよしなに改善する」ような広すぎる依頼になっていないこと
- タスク達成に必要な裁量と、プロダクト仕様を勝手に確定してはいけない境界が分かれていること
- 編集可能範囲、閲覧可能範囲、報告だけに留める範囲が混同されていないこと

## 完了条件がレビュー可能であること

- AI が何を出力・生成・修正・判定すれば完了なのかが明示されていること
- 人間または caller 側処理が、成果物をレビューできる粒度で完了条件が書かれていること
- 成功時に返すべき情報、未完了時に返すべき情報、失敗時に返すべき情報が区別されていること
- 「よい感じにする」「適切に処理する」だけでなく、少なくとも対象・観点・制約・出力形式が具体化されていること
- 作業結果に残る risk、assumption、未検証事項を隠さず返すべき場合、その出力先または field が明示されていること

## 不確実性・矛盾・不足情報の扱いが明確であること

- 根拠不足の場合に、推測で埋めてよいのか、不足情報として返すべきなのかが明示されていること
- AI が仮定を置いてよい場合、その仮定をどこに記録するかが明示されていること
- oracle と他の入力が衝突した場合に、どちらへ従うか、または衝突として報告するかが明示されていること
- 矛盾を見つけたときに、勝手に正本を修正してよいかどうかが誤解されないこと
- 候補なし、該当なし、判断不能、権限不足を、成功結果と混同しない出力にできること

## 出力形式と意味論が一致していること

- Structured Output を要求する場合、schema に適合するだけでなく、各 field の意味論が prompt 上で明確であること
- schema validation で検出できる制約と、AI に意味的判断として守らせる制約が分離されていること
- caller 側で変わる値を、schema 付属プロンプトなど複数箇所に手書きしていないこと
- Markdown 出力を要求する場合、見出し・箇条書き・コードブロックの意味が AI に誤解されないこと
- 「schema 外の説明を返すな」など、呼び出し側が機械処理するために必要な出力制約が明示されていること

## セキュリティ境界と prompt injection 耐性が保たれていること

- data として読む入力の中に命令文が含まれていても、それに従わないことが結合後プロンプトから読み取れること
- ユーザー入力、既存ファイル、ログ、検索結果、AI 生成済み中間成果物が、固定プロンプトや task-specificrules を上書きできないこと
- 読み取り禁止・編集禁止の path に対して、存在確認や内容推測を誘発する指示になっていないこと
- 権限変更、制約解除、別タスクへの誘導が data 側に含まれる場合でも、制御指示として扱わないこと
- 安全・アクセス制限に関する固定プロンプトの優先順位が、後続ブロックによって弱められていないこと
