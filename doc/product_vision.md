# `ticket_guys_b_team` Product Vision

## 1. 文書の目的

本書は `ticket_guys_b_team` の背景・目的・設計思想を説明する文書である。

本書は詳細仕様書ではない。状態遷移、ファイル形式、CLI 契約、Codex CLI wrapper 契約などの実装判断に必要な詳細は、分割された仕様文書を参照する。

---

## 2. プロダクト概要

`ticket_guys_b_team` は、仕様レビュー駆動の AI 開発支援を、人間が扱いやすい形に整理して実行するための CLI フロントエンドである。

本プロダクトの中心課題は、エージェントそのものの性能ではない。むしろ、以下の流れを破綻しにくく・追跡しやすく・人間が監督しやすい形へ整理することにある。

* 曖昧な要望を Plan へ落とすこと
* 人間が Plan をレビューし、修正し続けられること
* Plan から必要 Ticket を AI が生成すること
* Ticket を依存関係つきで実行すること
* 実行結果を Ticket に要約として残すこと
* 実行結果を踏まえてフォローアップ Ticket を追加できること
* 本番用 live 実行と、テスト用 stub 実行を同一業務モデルの上で切り替えられること

MVP では、外から見える Ticket 種別は worker 1 種に絞る。内部的には実装・チェック・リファクタ等のテンプレート差はあり得るが、利用者へ複数 Ticket 種別を露出しない。

---

## 3. 解決したい問題

ソフトウェア開発へ AI エージェントを導入するとき、問題になりやすいのは実装能力そのものよりも、運用の曖昧さである。

典型的には以下のような問題が起きる。

* 要望が曖昧なまま下流へ流れる
* 仕様レビューを飛ばして実装が始まる
* 実行単位の切り方が曖昧で、どこから再開すべきか分からない
* その時点で処理を打ち切ってよいのか分からない
* 実行失敗時に、何が起きたか追跡できない
* `codex exec` への依存が強すぎて、テストが重い、遅い、高コストになる
* 過去の実行結果を再現したいが、入力と出力が実行基盤の内部に埋もれてしまう
* 並列実行によって順序依存の差分が発生し、再現性が失われる

`ticket_guys_b_team` は、これらを「エージェントが賢く解いてくれること」に期待するのではなく、Plan、Ticket、状態、成果物、実行記録を構造化することによって抑制する。

---

## 4. 基本思想

### 4.1 上流で意図を固定する

本プロダクトは、「まず走らせて後で人間が見る」思想を取らない。

むしろ以下を優先する。

* 上流で Plan を固める
* 人間が Plan をレビューする
* 下流では Plan と Ticket 群を見ながら AI が次の作業単位を切る

### 4.2 独立した `approve` を持たない

本仕様では、Plan 承認のための独立サブコマンドを持たない。

理由は、Plan 承認と Ticket 生成を別イベントにすると、Plan 状態と Ticket 集合の整合性を壊しやすいからである。

代わりに、人間は `tgbt plan` で Plan を編集し、実行に進める意思決定は `tgbt run --plan-id ...` によって表す。

つまり、MVP では「run を開始すること」が運用上の承認に相当する。

### 4.3 Ticket は Plan 実行中に増減してよい

Plan から最初に固定枚数の Ticket を切り出し、それを消化したら終わり、という前提は置かない。

むしろ、`run` の最中に以下を反復する。

* 現在のリポジトリ状態と既存 Ticket 群を見る
* 必要なら新しい Ticket を作る
* 実行可能な Ticket を実行する
* 実行結果からフォローアップ Ticket を作る
* 既に役目を終えた Ticket を `settled` にする

これにより、初回分解の誤りや不足を run 中に自然に吸収しやすくする。

### 4.4 人間は意味を判断し、AI は処理を回す

人間の主責務は、意味と意図の判断である。
AI の主責務は、Plan 草案化、Ticket 生成、Ticket 実行、実行結果要約、フォローアップ Ticket 生成、記録整理である。

### 4.5 実行基盤の概念を直接見せない

内部ではエージェントランタイムやワークフローエンジンを利用していても、利用者にはそのまま露出しない。

利用者が触るのは以下の業務概念である。

* plan
* ticket
* run

live / stub の差異も、利用者にとっては「Codex 呼び出しの実行ソースが異なる」だけであり、Plan や Ticket の意味そのものは変わらない。

### 4.6 本番実行と再現実行を分離する

本プロダクトは、live 実行と stub 実行を同じものとして曖昧に扱わない。

* live は、本当に `codex exec` を呼び出す本番系の経路である
* stub は、過去の session record を strict replay するテスト系の経路である

重要なのは、業務上の状態遷移は共通のまま、外部依存だけを差し替えられることである。

### 4.7 実記録は再利用可能な資産として扱う

live 実行で発生した session record は、単なる監査ログではなく、stub 実行へ転用可能な資産として扱う。

これにより、以下を実現しやすくする。

* 高コスト実行の再現
* 回帰テスト
* バグ再現
* CLI 層から wrapper 層までの結合テスト
* 実運用で得たパターンの fixture 化

### 4.8 直列実行のみを採る

本プロダクトは、並列 Ticket 実行を採用しない。

理由は以下のとおりである。

* Ticket 生成と follow-up 生成が run の途中で起きるため、実行順序差が業務結果差に直結する
* 単一リポジトリ上での並列変更は、監査証跡の解釈を難しくする
* strict replay は wrapper 呼び出し列が全順序であることと相性がよい
* 再現性と追跡可能性を主価値に置く本プロダクトの基本思想と、並列化による非決定性が衝突する

したがって、本仕様では run と wrapper 呼び出しを将来的にも直列実行のみとする。

---

## 5. プロダクトが大事にする価値

### 5.1 明確さ

* Plan の状態が明確であること
* Plan revision が明確であること
* Ticket の現在状態が明確であること
* Ticket 依存から実行可否が判定できること
* Ticket 実行結果が Ticket 自身に要約として残ること

### 5.2 追跡可能性

* 何を入力に何を実行したか追跡できること
* 成功も失敗もログに残ること
* 成果物の位置が分かること
* live 実行の記録を stub 再生元として再利用できること
* どの Ticket / run / call がどの `plan_revision` に属していたか復元できること

### 5.3 分離

* 仕様と実装を分離する
* 実装と実行基盤を分離する
* 背景説明と厳密仕様を分離する
* CLI 層と `codex exec` 直接呼び出しを分離する

### 5.4 安全な自動化

* 未整理の要望から直接実装へ飛ばないこと
* Ticket 依存未解決のまま走らないこと
* run 中に作業単位を作り直せること
* 失敗時にも状態と記録を残すこと
* 同一 Plan に対する同時 run を禁止すること

### 5.5 再現可能性

* 同じ業務入力に対し、stub で過去の live 結果を再現できること
* テスト時に外部トークン消費や外部依存を避けられること
* wrapper 境界で入出力の差を局所化できること
* counters と front matter を復元すれば、同じ `run_id` / `codex_call_id` 系列を再生できること

---

## 6. 対象範囲

初期バージョンで対象とするのは、単一リポジトリに対する CLI ベースの実行支援である。

主に以下を扱う。

* 人間要望からの Plan 草案生成と更新
* `run --plan-id` による Plan 実行開始
* Plan からの Ticket 生成
* Ticket 実行の統制
* 実行結果サマリーの Ticket への反映
* フォローアップ Ticket 生成
* `codex exec` をラップする wrapper 経由の実行
* live 記録の保存
* strict replay による再現実行
* 実行結果とログ、成果物パスの整理

---

## 7. 対象外

初期バージョンでは、以下は主対象としない。

* Web UI
* 複数リポジトリ横断実行
* 分散ワーカー実行
* 並列 Ticket 実行
* 汎用エージェントプラットフォーム化
* エージェントランタイムの詳細設定を人間へ露出すること
* `run` にぶら下がる dry-run / preflight / validate 専用モード
* 専用の `status` / `state` 集約コマンド
* review / integration / root といった補助 Ticket 型
* 自動受け入れゲートの厳密仕様
* 任意 record を割り当てる manifest ベース replay
* あらゆる外部ツールを一般化した replay framework

MVP では、まず Plan / Ticket / run / wrapper の整合性を固める。自動受け入れゲートは将来拡張として別途仕様化する。

---

## 8. 想定ユーザー体験

理想的な利用体験は次のようなものである。

1. 人間が曖昧な要望を書く
2. AI が Plan 草案を生成する
3. 人間が Plan をレビューし、必要なだけ修正する
4. 人間が `run --plan-id` を開始する
5. AI が Plan と既存 Ticket 群を見て必要 Ticket を作る
6. AI が依存関係に従って Ticket を直列実行する
7. AI が Ticket ごとの実行結果を要約として書き戻す
8. AI が必要ならフォローアップ Ticket を追加する
9. すべての active Ticket が `settled` になり、追加 Ticket が不要になったら Plan が `settled` になる

ここで重要なのは、人間を外すことではなく、人間の判断を本当に必要な箇所へ圧縮することである。

---

## 9. 人間と AI の役割分担

### 人間

* Plan のレビューと修正指示
* `run` を開始する意思決定
* 機械的に判定できない例外判断
* live 記録を fixture として採用するかの判断
* 最終意思決定

### AI

* Plan 草案生成
* Ticket 生成と更新
* Ticket 実行
* 実行結果サマリーの作成
* フォローアップ Ticket 生成
* ログと成果物の整理
* live 記録の保存と stub 再生の実行
