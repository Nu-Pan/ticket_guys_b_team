# `ticket_guys_b_team` Product Vision

## 1. 文書の目的

本書は `ticket_guys_b_team` の背景・目的・設計思想を説明する文書である。

本書は詳細仕様書ではない。状態遷移、ファイル形式、CLI 契約、Codex CLI wrapper 契約などの実装判断に必要な詳細は、分割された仕様文書を参照する。

本書が扱うのは以下である。

* なぜこのプロダクトを作るのか
* 何を解決対象とするのか
* どのような思想で設計するのか
* 人間と AI の役割をどう分けるのか
* 初期バージョンでどこまでを目指すのか
* live / stub をなぜ設けるのか

---

## 2. プロダクト概要

`ticket_guys_b_team` は、仕様レビュー駆動の AI 開発支援を、人間が扱いやすい形に整理して実行するための CLI フロントエンドである。

本プロダクトの中心課題は、エージェントそのものの性能ではない。むしろ、以下のような上流から下流までの運用を、破綻しにくく・追跡しやすく・人間が監督しやすい形へ整理することにある。

* 曖昧な要望を Plan へ落とすこと
* Plan を人間がレビューし、承認可能な形へ磨くこと
* 承認済み Plan から Ticket を用意すること
* Ticket を依存関係つきで実行すること
* 実行結果を自動受け入れゲート中心で判定すること
* 機械判定不能な論点だけを人間へ戻すこと
* 本番用 live 実行と、テスト用 stub 実行を同一業務モデルの上で切り替えられること

MVP では、Ticket は worker 1 種に絞る。review / integration / root のような補助的 Ticket は導入しない。必要になるまでモデルを増やさず、Plan、worker Ticket、run の 3 つに集中する。

---

## 3. 解決したい問題

ソフトウェア開発へ AI エージェントを導入するとき、問題になりやすいのは実装能力そのものよりも、運用の曖昧さである。

典型的には以下のような問題が起きる。

* 要望が曖昧なまま下流へ流れる
* 仕様レビューを飛ばして実装が始まる
* 完了条件が曖昧で、終わったのか分からない
* 人間レビュー前提でしか安全性を確保できない
* 実行失敗時に、何が起きたか追跡できない
* エスカレーション条件が曖昧で、AI が進んでよい範囲が定まらない
* `codex exec` への依存が強すぎて、テストが重い、遅い、高コストになる
* 以前の live 実行を再現したいが、入力と出力が実行基盤の内部に埋もれてしまう

`ticket_guys_b_team` は、これらを「エージェントが賢く解いてくれること」に期待するのではなく、仕様・状態・成果物・検証を構造化することによって抑制する。

---

## 4. 基本思想

### 4.1 上流で意図を固定する

本プロダクトは、「まず走らせて後で人間が見る」思想を取らない。

むしろ以下を優先する。

* 上流で仕様を固める
* 人間がレビューする
* 下流は受け入れ条件に基づいて機械的に流す

つまり、創発性よりも統制を優先する。

### 4.2 人間は意味を判断し、AI は処理を回す

人間の主責務は、意味と意図の判断である。
AI の主責務は、草案化・実行・検証・記録である。

### 4.3 実行基盤の概念を直接見せない

内部ではエージェントランタイムやワークフローエンジンを利用していても、利用者にはそのまま露出しない。

利用者が触るのは以下のような業務概念である。

* plan
* ticket
* approve
* run

live / stub の差異も、利用者にとっては worker の実行ソースが異なるだけであり、Plan や Ticket の意味そのものは変わらない。

### 4.4 自動受け入れゲートを中心に据える

本プロダクトは、下流処理を「人間の全件レビュー」で安全化することを目指さない。

代わりに、以下のような観測可能な判定を重視する。

* テスト
* 静的検査
* 成果物存在確認
* 状態確認
* ログ確認

人間判断が必要なものは例外として扱い、`blocked` として明示的に止める。

### 4.5 本番実行と再現実行を分離する

本プロダクトは、live 実行と stub 実行を同じものとして曖昧に扱わない。

* live は、本当に `codex exec` を呼び出す本番系の経路である
* stub は、過去記録または用意された fixture を返すテスト系の経路である

重要なのは、業務上の状態遷移は共通のまま、外部依存だけを差し替えられることである。

### 4.6 実記録は再利用可能な資産として扱う

live 実行で発生した入出力は、単なる監査ログではなく、stub 実行へ転用可能な資産として扱う。

これにより、以下を実現しやすくする。

* 高コスト実行の再現
* 回帰テスト
* バグ再現
* CLI 層から wrapper 層までの結合テスト
* 実運用で得たパターンの fixture 化

---

## 5. プロダクトが大事にする価値

### 5.1 明確さ

* Plan の状態が明確であること
* Ticket の完了条件が明確であること
* 実行可否が依存関係から明確に判定できること

### 5.2 追跡可能性

* 何を入力に何を実行したか追跡できること
* 成功も失敗もログに残ること
* 成果物の位置が分かること
* live 実行の記録を stub 再生元として明示指定できること

### 5.3 分離

* 仕様と実装を分離する
* 実装と実行基盤を分離する
* 背景説明と厳密仕様を分離する
* CLI 層と `codex exec` 直接呼び出しを分離する

### 5.4 安全な自動化

* 未承認仕様から先へ進まないこと
* 依存未解決のまま走らないこと
* 判定不能なら勝手に進めないこと
* 失敗時にも状態と記録を残すこと

### 5.5 再現可能性

* 同じ業務入力に対し、stub で過去の live 結果を再現できること
* テスト時に外部トークン消費や外部依存を避けられること
* wrapper 境界で入出力の差を局所化できること

---

## 6. 対象範囲

初期バージョンで対象とするのは、単一リポジトリに対する CLI ベースの実行支援である。

主に以下を扱う。

* 人間要望からの Plan 草案生成
* Plan レビューと承認の支援
* 承認済み Plan からの worker Ticket 生成または更新
* worker Ticket 実行の統制
* `codex exec` をラップする wrapper 経由の worker 実行
* live 記録の保存
* stub による再現実行
* 実行結果とログ、成果物パスの整理

---

## 7. 対象外

初期バージョンでは、以下は主対象としない。

* Web UI
* 複数リポジトリ横断実行
* 分散ワーカー実行
* 汎用エージェントプラットフォーム化
* エージェントランタイムの詳細設定を人間へ露出すること
* `run` にぶら下がる dry-run / preflight / validate 専用モード
* 専用の `status` / `state` 集約コマンド
* review / integration / root といった補助 Ticket 型
* 完全自動な fixture 正規化パイプライン
* あらゆる外部ツールを一般化した replay framework

ファイルが唯一の正本であり、MVP では状態確認のための専用コマンドを増やさない。必要になった場合のみ将来追加する。

---

## 8. 想定ユーザー体験

理想的な利用体験は次のようなものである。

1. 人間が曖昧な要望を書く
2. AI が Plan 草案を生成する
3. 人間が Plan をレビューし、承認する
4. AI が Plan から worker Ticket を生成または更新する
5. AI が依存関係に従って Ticket を実行する
6. worker 実行では live または stub を選択できる
7. 自動受け入れゲートが結果を判定する
8. 機械判定不能な論点だけが人間へ戻る
9. 人間は必要最小限の判断だけを行う

ここで重要なのは、人間を外すことではなく、人間の判断を本当に必要な箇所へ圧縮することである。

---

## 9. 人間と AI の役割分担

### 人間

* Plan のレビュー
* Plan の承認
* 機械的に判定できない例外判断
* live 記録を fixture として採用するかの判断
* 最終意思決定

### AI

* 草案生成
* worker Ticket 生成または更新
* worker Ticket 実行
* 自動受け入れゲート実行
* エスカレーション論点の抽出
* ログと成果物の整理
* live 記録の保存と stub 再生の実行
